# v0.2 Implementation Review: PRD vs. Actual

**Date:** 2026-04-06
**Scope:** All requirements in `docs/requirements-v0.2.md` compared against implemented code.

---

## Overview

The v0.2 Two-Stream Intake implementation was executed methodically across all 11 work areas (A–K) of the WBS. Architecture is clean, type safety is strong, and the vast majority of PRD requirements are met. This review identifies seven mismatches between the PRD intent and the shipped implementation, ranked by severity.

---

## Findings

### 1. CRITICAL — Review Action Routes to Wrong Folder on Arrival

**PRD §1.2:**

> During **arrival routing**, the message always goes to the Review folder **regardless of whether `folder` is set** — the folder field has no effect at arrival time.

**Implementation** (`src/actions/index.ts:42–44`):

```typescript
case 'review': {
  const folder = action.folder ?? ctx.reviewFolder;
  return executeMove(ctx.client, message, folder, base).then((r) => ({ ...r, action: 'review' }));
}
```

The code uses `action.folder` if present, falling back to the Review folder only when `folder` is absent. A `review` rule with `folder: "MailingLists"` moves the message directly to `MailingLists` on arrival instead of `Review`. The `folder` field on a review action is a sweep-time archive hint, not an arrival-time destination.

**Fix:** Replace with `const folder = ctx.reviewFolder;` — always use the Review folder at arrival time.

### 2. MODERATE — Sweep Logs Every Action with `rule: null`

**PRD §5.4:**

> `rule_id` / `rule_name`: the rule that determined the destination. This could be: (a) a move/delete rule that matched during re-evaluation, (b) the original review rule whose folder field was used, or (c) null if the global default archive folder was used.

**Implementation** (`src/sweep/index.ts:199`):

```typescript
this.activityLog.logActivity(result, emailMsg, null, 'sweep');
```

The rule is **always** `null`. The `resolveSweepDestination` function returns the destination folder but discards which rule matched. Users cannot trace why a message was archived to a particular folder.

**Fix:** Have `resolveSweepDestination` return the matched rule alongside the destination, and pass it through to `logActivity`.

---

### 3. MODERATE — Sweep Re-evaluation Doesn't Filter Out `review` Rules

**PRD §3.3** defines a three-step archive destination resolution:

1. Re-evaluate against `move`/`delete` rules only (skip `review` **and** `skip` actions)
2. If originating `review` rule has a `folder` field, use that folder
3. Fall back to `review.defaultArchiveFolder`

**Implementation** (`src/sweep/index.ts:20–21`):

```typescript
const candidates = rules.filter((r) => r.action.type !== 'skip');
```

Only `skip` rules are filtered out. `review` rules remain in the candidate set and match during re-evaluation, collapsing steps 1 and 2 into a single pass. The practical consequence: if a `review` rule (without a `folder`) matches first during re-evaluation, the message falls through to the global default — bypassing a lower-priority `move` rule that the PRD intends to be considered in step 1.

**Fix:** Filter both `review` and `skip` from the step-1 candidate set. If no `move`/`delete` rule matches, then separately find the originating `review` rule and check its `folder` field (step 2), then fall back to the global default (step 3).

### 4. MINOR — Trash Folder Resolution Skips IMAP Special-Use Detection

**PRD §9.1:**

> 1. Use the IMAP `\Trash` special-use attribute to find the correct folder name
> 2. If special-use detection fails, fall back to a configurable name: `review.trashFolder`

**Implementation** (`src/index.ts:38`):

```typescript
const trashFolder = config.review.trashFolder;
```

The trash folder is always read from config. The IMAP client has a `getSpecialUseFolder()` method but it is never called for trash resolution. Servers that name their trash folder "Deleted Items" or "Deleted Messages" will not be handled correctly unless the user manually sets `review.trashFolder`.

**Fix:** After IMAP connection, call `getSpecialUseFolder('\\Trash')` and fall back to `config.review.trashFolder` if it returns null.

---

### 5. MINOR — `trashFolder` Config Key Not in PRD Schema

**PRD §6.2** defines the review config as:

```yaml
review:
  folder: string
  defaultArchiveFolder: string
  sweep:
    intervalHours: number
    readMaxAgeDays: number
    unreadMaxAgeDays: number
```

The implementation adds `trashFolder` (default `"Trash"`) to the review config schema. This is a reasonable implementation choice given that the PRD does mention a fallback config key (`review.trashFolder` in §9.1), but it's worth noting the config shape diverges from the spec.

**Impact:** Low. The addition is additive and backward-compatible.

---

### 6. MINOR — Sweep Settings Not Editable in UI

**PRD §7.5:**

> Optionally editable inline — but at minimum, display them.

The sweep settings card in the Settings page is display-only. This meets the minimum bar. The backend plumbing exists (`PUT /api/config/review` is wired and functional), but the frontend does not expose edit controls for sweep thresholds, interval, or folder names.

**Impact:** Low. Meets minimum requirement; edit capability can be added later.

---

### 7. MINOR — Sweeper Reference in ServerDeps Goes Stale After Config Reload

**PRD §11.3:**

> Restarted when review config changes (new trigger).

When review config or IMAP config changes, `src/index.ts` creates a new `ReviewSweeper` and assigns it to the local `sweeper` variable. However, the `ServerDeps` object passed to `buildServer` (line 88–93) holds a reference to the **original** sweeper instance. After a config change, `GET /api/review/status` reads state from the old, stopped sweeper.

**Fix:** Use a mutable reference (e.g., a getter function or wrapper object) in `ServerDeps` so the route always reads from the current sweeper.

## Recommendation

Fix issues 1–3 before considering v0.2 complete. Issue 1 is a behavioral bug that sends mail to the wrong folder. Issues 2–3 undermine the sweep's auditability and resolution correctness. Issues 4–7 are papercuts that can be addressed at your leisure or deferred.
