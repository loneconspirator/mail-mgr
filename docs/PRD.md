# Mail Manager — Product Requirements Document

## Overview

Mail Manager is a self-hosted IMAP mailbox monitor that automatically classifies and files incoming email. It connects to a single IMAP account, monitors the inbox, and moves messages to appropriate folders based on a combination of deterministic rules and LLM-based classification.

The system is designed around two core principles:

1. **Layered rule evaluation** — Deterministic rules run first (fast, free, private). LLM classification is a fallback for what pattern matching can't handle.
2. **Privacy-focused progressive disclosure** — When LLM classification is needed, the system sends the minimum information necessary, escalating from envelope data to subject to body only as needed and only as permitted.

Each instance monitors one IMAP account. Multiple accounts are handled by running multiple instances. The application runs in Docker with a web UI for configuration and management. Rules are stored in a config file that the UI reads and writes.

## Rule System

### Rule Types

**Deterministic rules** — Pattern matches on sender, recipient, and/or subject. Evaluated first. No external API calls. Example: sender matches `*@github.com` and recipient is `mike+oss@example.com` → move to `Dev/OSS`.

**LLM-instructed rules** — English-language instructions provided by the user, evaluated by an LLM. Example: "Anything related to invoices goes to Finance." Subject to progressive disclosure. Evaluated only when no deterministic rule matches.

**Suggested rules** *(Tier 4)* — Generated by the system based on observed user filing behavior. These remain in a pending state until the user approves, modifies, or rejects them. Upon approval they become deterministic or LLM-instructed rules.

### Evaluation Order

1. Deterministic rules (fast, private, free)
2. LLM-instructed rules with progressive disclosure (slower, costs money, minimizes data exposure)
3. No match — message stays in the inbox

Recipient address is a first-class variable in all rule types, not secondary to sender matching.

## Privacy-Focused Progressive Disclosure

When an email requires LLM classification, the system reveals information to the LLM incrementally, stopping as soon as it gets a confident classification:

**Level 1 — Envelope only:** Sender address, recipient address(es). No subject, no body. Many emails can be classified purely on who sent them and which address received them.

**Level 2 — Plus subject:** If Level 1 doesn't produce a confident classification, add the subject line.

**Level 3 — Plus body text:** If still unclassified and permitted, send the plain text body. This is the most invasive level and requires explicit opt-in.

### Disclosure Controls

- **Global default** — Configurable maximum disclosure level applied to all messages (default: Level 3).
- **Filter-based caps** — Using the same sender/recipient/subject matching syntax as deterministic rules, the user can set a maximum disclosure level for specific patterns. Example: mail to `mike+personal@example.com` capped at Level 2 means the body of those emails never reaches the LLM.

The system logs which disclosure level was required for each classification, providing data to inform future rule creation and optimization.

## Actions

Rules can trigger the following actions:

- **Move to folder** — The primary action. Most rules do this.
- **Delete / Trash** — Remove the message. Useful for transient content.
- **No action / Skip** — Explicitly mark messages as "leave alone" to prevent repeated classification attempts.

### Conditional Actions

Rules can include conditions beyond content matching:

- **Message age** — e.g., "older than 30 minutes"
- **Read status** — e.g., "already read"
- **Combination** — e.g., "read OTP emails older than 30 minutes → trash"

Implementation of time-based conditions (scheduled follow-up vs. periodic sweep) is left as an implementation detail.

## Monitoring Scope

- **Inbox** — The primary monitored folder. All classification and rule evaluation happens here.
- **Other folders** *(Tier 4)* — Periodically scanned to observe where the user manually files messages, feeding the autonomous rule suggestion system.

## User Interface

A simple web UI served by the application, providing:

- Rule management (create, edit, delete, reorder)
- Recent activity log (what was moved, what rule matched, what disclosure level was used)
- IMAP connection configuration
- LLM API configuration *(Tier 2+)*
- Disclosure level settings *(Tier 2+)*
- Rule suggestion review *(Tier 4)*

## Deployment

- Packaged as a Docker container
- Single instance per IMAP account
- Configuration and rules stored in a config file (mounted as a volume or managed through the UI)

## Implementation Tiers

### Tier 1 — Deterministic Engine

The foundation. A working mail sorter with no LLM dependency.

- IMAP connection to a single mailbox, monitoring inbox
- IMAP IDLE for new message notification, with polling fallback
- Deterministic rules: match on sender, recipient, subject → move to folder
- Rules stored in a config file
- Web UI: manage rules, view recent activity/log, configure IMAP credentials
- Docker deployment

### Tier 2 — LLM Classification

Add intelligence. Handle the mail that deterministic rules can't.

- LLM classification with progressive disclosure (Levels 1–3)
- English-language LLM-instructed rules
- Global default disclosure level setting
- LLM API configuration in the web UI

### Tier 3 — Conditions and Privacy Controls

Refine behavior. Handle time-sensitive content and tighten privacy boundaries.

- Time and state-based rule conditions (message age, read status)
- Trash/delete actions
- Disclosure level cap filters (per sender/recipient/subject pattern)
- Deterministic pre-filters on LLM rules (gate LLM evaluation behind pattern matches)
- Classification confidence and disclosure level logging

### Tier 4 — Down the Road

The system gets smarter on its own. Details intentionally kept vague.

- Scan non-inbox folders to observe user filing behavior
- Autonomous rule suggestions based on observed patterns
- Suggestion review workflow in the UI (approve / modify / reject)
- Rule effectiveness analytics
- Suggested deterministic rule promotions (system notices an LLM rule could be a simple pattern match)
- Human-readable / portable rule file format

## Technology

Intentionally unspecified. The application could be built as a Python service, a Node.js app, an n8n workflow, or something else. The PRD describes behavior, not implementation. Technology decisions will be made when implementation begins.
