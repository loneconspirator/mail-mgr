### Navigation Architecture

Phase 15 adds two new pages to the existing `data-page` navigation:
- `data-page="reviewed"` -> renders Reviewed Senders view
- `data-page="archived"` -> renders Archived Senders view

These sit alongside the Phase 14 nav buttons (Priority, Blocked) in the header `<nav>`. Existing "Rules" page remains unchanged.

### Folder Group Layout

Each folder group is a collapsible accordion section:

```
[v] INBOX.MailingLists (3 senders)
   +-----------------------------------------+
   | Sender               | Rule Name        |
   |-----------------------|------------------|
   | *@newsletter.com      | Newsletter rule  |
   | *@updates.example.com |                  |
   | alerts@service.io     | Service alerts   |
   +-----------------------------------------+

[>] Review (1 sender)
   (collapsed — click to expand)
```

#### Folder Group Header Row

| Element | Content | Style |
|---------|---------|-------|
| Toggle arrow | `>` (collapsed) / `v` (expanded) | 16px wide, `#888` color, `.tree-toggle` pattern |
| Folder path | Full IMAP folder path (e.g., `INBOX.MailingLists`) | 14px semibold `#222` |
| Sender count | `(N senders)` or `(1 sender)` | 13px regular `#888`, right-aligned or inline after folder name |

#### Sender Table Within Group

| Column | Content | Width |
|--------|---------|-------|
| Sender | Sender pattern from rule match (e.g., `*@example.com`) | auto (flex) |
| Rule Name | Optional rule name, muted if absent | 200px max |

Same two-column layout as Phase 14 Priority/Blocked tables.

### Reviewed Senders: Default Folder Logic

- Rules with `action.type === 'review'` and `action.folder` specified: group under that folder path
- Rules with `action.type === 'review'` and NO `action.folder`: group under the default Review folder name (fetched from `GET /api/config/review` -> `folder` field, default `"Review"`)
- The default folder group header shows the resolved folder name (e.g., `Review`) not a placeholder

### Folder Group Sorting

- Folder groups sorted alphabetically by folder path
- Senders within each group sorted alphabetically by sender pattern

### Collapse State

- All folder groups start expanded by default
- Collapse/expand is toggled per-group by clicking the folder group header
- Collapse state is ephemeral (not persisted across navigations)

## Interaction Contracts

### Tab Navigation

1. User clicks "Reviewed" or "Archived" nav button in header
2. Active state transfers to clicked button (`.nav-btn.active`)
3. `#app` clears and renders the appropriate folder-grouped view
4. Fetches from `GET /api/dispositions?type={review|move}` on each navigate
5. For Reviewed view: also fetches `GET /api/config/review` to resolve default folder name
6. Loading state: `<p>Loading...</p>` (existing pattern)

### Reviewed Senders View (VIEW-03)

- Calls `GET /api/dispositions?type=review`
- Also calls `GET /api/config/review` to get default review folder name
- Groups returned rules by destination folder (`action.folder` or default review folder)
- Each folder group contains a table of sender rows
- Each row shows: sender pattern, optional rule name
- Empty state when no results

### Archived Senders View (VIEW-04)

- Calls `GET /api/dispositions?type=move`
- Groups returned rules by destination folder (`action.folder`)
- Each folder group contains a table of sender rows
- Each row shows: sender pattern, optional rule name
- Empty state when no results

### States

| State | Visual |
|-------|--------|
| Loading | `<p>Loading...</p>` centered in `#app` |
| Data loaded | Folder groups with nested sender tables, all expanded |
| Empty | `.empty` div with heading + guidance copy |
| Error | `.empty` div with error message (existing pattern) |
| Group collapsed | Header row visible, nested table hidden |
| Group expanded | Header row visible, nested table visible below |

## Shared Rendering Logic

Phase 15 introduces folder-grouped rendering that both Reviewed and Archived views share. The implementation should extract a shared function:

```
renderFolderGroupedView(rules: Rule[], heading: string, emptyConfig: {heading, body}, defaultFolder?: string)
```

This function:
1. Groups rules by `action.folder` (or `defaultFolder` if folder is absent)
2. Sorts groups alphabetically by folder name
3. Sorts senders within each group alphabetically
4. Renders the folder group accordion with nested sender tables
5. Shows empty state if no rules

The flat `renderDispositionView()` from Phase 14 remains unchanged for Priority/Blocked.

## Accessibility Contract

| Requirement | Implementation |
|-------------|----------------|
| Nav buttons | Use `<button>` elements (existing pattern), keyboard-focusable by default |
| Active tab | Visual indicator via `.nav-btn.active` class |
| Folder group headers | Use `<button>` or clickable `<div>` with `role="button"` and `tabindex="0"` for keyboard access |
| Expand/collapse | `aria-expanded="true|false"` on folder group header buttons |
| Group content | `aria-hidden` toggled to match collapse state, or DOM removal on collapse |
| Table headers | Use `<th>` elements with descriptive text |
| Empty states | Semantic text content, no ARIA needed for simple text |
