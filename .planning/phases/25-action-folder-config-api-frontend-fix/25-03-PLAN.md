**Frontend fix (src/web/frontend/app.ts line 1668):**

The current guard checks `folderPath.startsWith(actionPrefix + '/')` which fails when IMAP uses `.` as delimiter. The file already has delimiter-sniffing logic at lines 1682-1686 but it runs AFTER the guard.

Move the delimiter detection BEFORE the action folder guard. Replace the single-delimiter check at line 1668 with a check that tests both common delimiters:

```typescript
const isActionFolder = folderPath === actionPrefix
  || folderPath.startsWith(actionPrefix + '/')
  || folderPath.startsWith(actionPrefix + '.');
```

This is simpler and more robust than trying to detect the delimiter first — it just checks both.

**Backend fix (src/web/routes/folders.ts line 51):**

When `findNode(tree, oldPath)` returns null (stale cache), the delimiter falls back to `'/'`. This means the backend action folder guard at line 65 (`oldPath.startsWith(actionPrefix + delimiter)`) also fails for dot-delimited paths.

Fix the fallback: instead of hardcoding `'/'`, detect the delimiter from `oldPath` itself. After line 51, replace:
```typescript
const delimiter = selectedNode?.delimiter || '/';
```
with:
```typescript
const delimiter = selectedNode?.delimiter || (oldPath.includes('.') ? '.' : '/');
```

This ensures the backend guard `oldPath.startsWith(actionPrefix + delimiter)` works even when tree cache is stale.

Add a test case: when oldPath is `"Actions.⭐ VIP Sender"` (dot-delimited, no matching tree node so findNode returns null), the route should return 403 with "System folders cannot be renamed". Use a mock tree that does NOT contain this folder path (simulating stale cache) so findNode returns null and delimiter fallback kicks in.

<task type="auto">
  <name>Task 2: Fix cold start crash from null-vs-undefined monitor rebuild</name>
  <files>src/index.ts, src/monitor/index.ts</files>
  <action>
**Root cause:** At src/index.ts line 274, `initialHeader` is `null` (from probeEnvelopeHeaders returning null) and `config.imap.envelopeHeader` is `undefined` (not set in config). The strict inequality `null !== undefined` evaluates to `true`, triggering an unnecessary monitor rebuild. The rebuild calls `monitor.stop()` which calls `this.client.disconnect()` which nullifies `this.flow` via `cleanupFlow()`. Then line 283 calls `imapClient.getSpecialUseFolder('\\Trash')` which throws "Not connected" because `this.flow` is null.

**Fix in src/index.ts line 274:**

Normalize both values before comparison so that null and undefined are treated as equivalent (both mean "no header discovered/configured"). Change:

```typescript
if (initialHeader !== config.imap.envelopeHeader) {
```

to:

```typescript
if ((initialHeader ?? undefined) !== config.imap.envelopeHeader) {
```

This coerces `null` to `undefined` so the comparison `undefined !== undefined` is `false`, skipping the unnecessary rebuild. When there IS an actual change (e.g., `"X-Original-To"` vs `undefined`), the rebuild still triggers correctly.

**Also add a defensive reconnect in monitor.stop() (src/monitor/index.ts lines 105-108):**

The current `stop()` method calls `this.client.disconnect()` which is destructive — it logs out the IMAP connection and nullifies the flow. Since the monitor shares the ImapClient with other consumers (action folder poller, trash resolution, etc.), disconnecting is too aggressive for a "rebuild" scenario.

Change `stop()` to only remove listeners and stop polling/idle, without disconnecting:

```typescript
async stop(): Promise<void> {
  this.client.removeAllListeners();
  // Do NOT disconnect — client is shared with other consumers
  // (action folder poller, getSpecialUseFolder, etc.)
  // Caller is responsible for client lifecycle.
}
```

Remove the `await this.client.disconnect()` call entirely. The monitor does not own the client — it receives it via dependency injection. The client lifecycle is managed by index.ts (which calls `imapClient.connect()` at startup and could call `disconnect()` at shutdown).

This is the correct fix because:
1. Monitor constructor receives the client, it doesn't create it
2. Other code (action folder poller, trash resolution) uses the same client after monitor.stop()
3. The shutdown path in index.ts can handle its own client disconnect if needed
