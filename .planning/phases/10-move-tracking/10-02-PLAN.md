    export interface MoveTrackerDeps {
      client: ImapClient
      activityLog: ActivityLog
      signalStore: SignalStore

      listId?: string

    **MoveTracker class:**

    Private state:
    - `scanTimer: ReturnType<typeof setInterval> | null = null`
    - `deepScanTimer: ReturnType<typeof setInterval> | null = null`
    - `running = false` (prevents overlapping scans)
    - `pendingConfirmation: Map<string, TrackedMessage>` -- messages missing in scan N, awaiting scan N+1 confirmation (keyed by `folder:uid`)
    - `signalsLogged = 0`

    **`start(): void`** (per D-01, D-10)
    If `!this.enabled`, return immediately.
    Call `this.stop()` first.
    Run `this.runScan()` immediately (fire-and-forget with `.catch()`).
    Set `this.scanTimer = setInterval(() => this.runScan().catch(...), this.scanIntervalMs)`.
    Set `this.deepScanTimer = setInterval(() => this.runDeepScan().catch(...), 15 * 60 * 1000)` per D-05.

    **`stop(): void`**
    Clear scanTimer, clear deepScanTimer.

    **`getState(): MoveTrackerState`**
    Return current status snapshot.

    **`private async runScan(): Promise<void>`**
    If `this.running` or `this.deps.client.state !== 'connected'`, skip.
    Set `this.running = true`, wrap in try/finally to reset.

    For each folder in `[this.deps.inboxFolder, this.deps.reviewFolder]`:
    1. **Fetch current UIDs with envelope data** using `this.deps.client.withMailboxLock(folder, async (flow) => { ... })`:
       - Use `flow.fetch('1:*', { uid: true, envelope: true, flags: true, headers: true }, { uid: true })` -- the `headers` fetch gets List-Id, the envelope gets sender/recipients/subject
       - Build a `Map<number, TrackedMessage>` from the results
       - Capture UIDVALIDITY from the mailbox info (check `flow.mailbox?.uidValidity` or similar ImapFlow property after lock acquisition)
    2. **Load previous snapshot** via `this.deps.activityLog.getState(\`tracking:${folder}:snapshot\`)`
       - Parse JSON as `FolderSnapshot | null`
    3. **UIDVALIDITY check** (per Pitfall 1):
       - If previous snapshot exists and `prevSnapshot.uidValidity !== currentUidValidity`, discard old snapshot, save current as new baseline, skip diffing for this folder
    4. **First-run baseline**:
       - If no previous snapshot, save current snapshot, skip diffing
    5. **Diff: find disappeared UIDs**:
       - Build Set of current UIDs
       - For each message in previous snapshot, if UID not in current set, it "disappeared"
    6. **Cross-reference activity log** (per LEARN-01):
       - For each disappeared message, query: `SELECT 1 FROM activity WHERE message_id = ? AND success = 1 AND source IN ('arrival', 'sweep', 'batch') AND timestamp > datetime('now', '-1 day') LIMIT 1`
       - NOTE: This requires adding a method to ActivityLog or passing the db directly. **Recommended approach**: Add a method `isSystemMove(messageId: string): boolean` to the MoveTracker class itself that queries the activity table via the same db instance. OR, simpler: just add the query inline using a prepared statement on the db. Since ActivityLog doesn't expose its db, the cleanest approach is to add a public method `ActivityLog.isSystemMove(messageId: string): boolean` that runs this query. Add this method to ActivityLog in this task.
       - If system move found, skip this message
    7. **Two-scan confirmation** (per Pitfall 2):
       - If message is NOT in `pendingConfirmation`, add it and skip (first detection)
       - If message IS in `pendingConfirmation` (second consecutive detection), proceed to destination resolution
    8. **Destination resolution**:
       - Call `this.deps.destinationResolver.resolveFast(msg.messageId, folder)`
       - If destination found, call `this.deps.signalStore.logSignal({ messageId: msg.messageId, sender: msg.sender, envelopeRecipient: msg.envelopeRecipient, listId: msg.listId, subject: msg.subject, readStatus: msg.readStatus, visibility: msg.visibility, sourceFolder: folder, destinationFolder: destination })`
       - If not found, call `this.deps.destinationResolver.enqueueDeepScan(msg.messageId, folder)` and keep in pendingConfirmation for deep scan resolution
    9. **Clean up pendingConfirmation**: Remove entries for messages that reappeared (are in current UID set)
    10. **Save snapshot**: `this.deps.activityLog.setState(\`tracking:${folder}:snapshot\`, JSON.stringify(currentSnapshot))`
    11. Update `lastScanAt`

    **`private async runDeepScan(): Promise<void>`**
    Call `this.deps.destinationResolver.runDeepScan()`.
    For each resolved messageId -> destinationFolder:
    - Look up the TrackedMessage from pendingConfirmation
    - Call `this.deps.signalStore.logSignal(...)` with full signal data
    - Remove from pendingConfirmation
    Per D-06: Messages not resolved by deep scan are dropped by DestinationResolver.runDeepScan() -- they just disappear from pendingDeepScan.

    **ActivityLog addition (src/log/index.ts):**
    Add a public method:
    ```typescript
    isSystemMove(messageId: string): boolean {
      const row = this.db.prepare(
        `SELECT 1 FROM activity WHERE message_id = ? AND success = 1
         AND source IN ('arrival', 'sweep', 'batch')
         AND timestamp > datetime('now', '-1 day') LIMIT 1`
      ).get(messageId)
      return row !== undefined
    }
    ```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| IMAP envelope data -> TrackedMessage | Untrusted email headers cached in snapshot and used in signal creation |
| IMAP UIDVALIDITY -> snapshot logic | Server-provided value controls whether snapshots are trusted |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-10-04 | Tampering | ActivityLog.isSystemMove() | mitigate | Parameterized query with `?` placeholder for messageId. No string interpolation. |
| T-10-05 | Spoofing | UIDVALIDITY from IMAP server | accept | UIDVALIDITY is provided by the IMAP server which we already trust for all operations. A compromised server could cause false re-baselines (lost tracking) but not false signals. Acceptable risk for single-instance app. |
| T-10-06 | Denial of Service | Large folder UID fetch | accept | UID-only fetch is lightweight even for 10K+ messages. withMailboxLock() serializes access. Scan interval (30s) is configurable upward if performance is an issue. |
</threat_model>
