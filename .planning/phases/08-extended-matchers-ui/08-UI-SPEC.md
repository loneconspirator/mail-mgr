## Interaction Contract

### Rule Modal — New Fields

| Interaction | Behavior |
|-------------|----------|
| Open modal (new rule) | All three new fields default to blank/unset. Delivered-To and Recipient Field are disabled if envelope header is unavailable. Read Status is always enabled. |
| Open modal (edit rule) | Fields pre-populated from saved rule values. If envelope unavailable, Delivered-To and Recipient Field show saved values in disabled state (D-13). |
| Delivered-To input | Text input with glob syntax. Placeholder: `*@example.com`. Disabled + grayed when envelope unavailable (D-11). |
| Recipient Field dropdown | Options: em-dash (blank), Direct, CC, BCC, List. Disabled + grayed when envelope unavailable (D-11). |
| Read Status dropdown | Options: em-dash (blank), Read, Unread. Always enabled (D-12). |
| Disabled field info icon | Unicode info mark (U+2139) after label text. HTML `title` attribute tooltip: "Envelope header not discovered -- run discovery in IMAP settings." |
| Save — value collection | Read all field values including disabled fields. Convert empty string to undefined before building match object. |
| Save — validation | At least one of: sender, subject, deliveredTo, visibility, readStatus must be non-empty. Error toast: "At least one match field is required". |
| Save — empty select handling | Empty `<select>` value `""` must be omitted (not sent as empty string) to avoid Zod enum validation rejection. |

### Settings Page — Envelope Discovery

| Interaction | Behavior |
|-------------|----------|
| Page load | Fetch envelope status via `api.config.getEnvelopeStatus()` in parallel with IMAP config and connection status. |
| Header discovered | Display: status badge (.connected) with header name (e.g., "Delivered-To") + text "detected" + "Re-run Discovery" button (plain .btn). |
| Header not discovered | Display: warning paragraph (.discovery-warning) with text "No envelope header detected. Rules using Delivered-To and Recipient Field will be skipped." + "Run Discovery" button (.btn.btn-primary). |
| Click discovery button | Button disables, text changes to spinner + "Discovering...", pointer-events: none, opacity: 0.7 (D-10). |
| Discovery success (header found) | Toast: "Discovered: {headerName}". Call renderSettings() to re-render entire page with updated status. |
| Discovery success (no header) | Toast (error style): "No envelope header found". Call renderSettings() to re-render. |
| Discovery error | Toast (error style) with error message. Restore button to original state (re-enable, remove spinner, restore text). |
| Discovery 409 conflict | Backend rejects concurrent discovery calls. Toast: error message from server. Button restored. |

### Behavior Description (Rule List)

| Interaction | Behavior |
|-------------|----------|
| Rule list rendering | Replace inline matchStr with `generateBehaviorDescription(rule.match)` from rule-display.ts. |
| Description format | Comma-separated parts: `sender: {glob}`, `subject: {glob}`, `delivered-to: {glob}`, `field: {value}`, `status: {value}`. Only non-empty fields included. |

## Field Order (Rule Modal)

Exact field order per D-01:

1. Name — `<input>` text, id `m-name`
2. Match Sender — `<input>` text, id `m-sender`, placeholder `*@example.com`
3. Match Subject — `<input>` text, id `m-subject`, placeholder `*newsletter*`
4. **Delivered-To** — `<input>` text, id `m-deliveredTo`, placeholder `*@example.com` (NEW)
5. **Recipient Field** — `<select>`, id `m-visibility` (NEW)
6. **Read Status** — `<select>`, id `m-readStatus` (NEW)
7. Action — `<select>`, id `m-action-type`
8. Folder — folder picker (conditional on action type)

No grouping, no separators, no collapsible sections per D-01.
