Modify `src/action-folders/processor.ts` with three changes:

**Change 1 (D-05): Post-move activity logging**

Update `buildActionResult` signature to accept a `success` parameter with default `true`:
```typescript
private buildActionResult(
  message: EmailMessage,
  action: string,
  ruleId: string,
  folder: string,
  success: boolean = true,
): ActionResult {
  return {
    success,
    messageUid: message.uid,
    messageId: message.messageId,
    action,
    folder,
    rule: ruleId,
    timestamp: new Date(),
  };
}
```

Restructure `processMessage` so ALL `logActivity` calls happen AFTER the `moveMessage` call succeeds. The pattern for each path:

1. **Create path (new rule):** Do business logic (conflict resolution + rule creation) -> accumulate activity entries as data (not logged yet) -> moveMessage -> log all accumulated activities -> return
2. **Duplicate path:** Detect duplicate -> moveMessage -> logActivity -> return (D-06)
3. **Remove path:** Delete rule -> moveMessage -> logActivity -> return
4. **No-match undo path:** moveMessage -> return (no activity to log)

For move failure in each path, log with `success: false`:
```typescript
try {
  await this.client.moveMessage(message.uid, destination, sourceFolder);
} catch (err) {
  this.logger.error({ uid: message.uid, err }, 'Failed to move message after action folder processing');
  // Log any pending activities with success: false
  for (const pending of pendingActivities) {
    const result = this.buildActionResult(message, pending.action, pending.ruleId, destination, false);
    this.activityLog.logActivity(result, message, pending.rule, 'action-folder');
  }
  return { ok: false, action: actionType, error: 'Message move failed' };
}
// Log all pending activities with success: true
for (const pending of pendingActivities) {
  const result = this.buildActionResult(message, pending.action, pending.ruleId, destination, true);
  this.activityLog.logActivity(result, message, pending.rule, 'action-folder');
}
```

Use a `pendingActivities` array typed as `Array<{ action: string; ruleId: string; rule: Rule }>` to accumulate activities before the move. This handles the conflict resolution case where there are 2 activities (removal + creation).

**Change 2 (D-06): Duplicate path early return**

The duplicate detection path must NOT fall through to the shared moveMessage at the bottom. Instead, handle its own move + log + return:

```typescript
if (duplicate) {
  this.logger.debug({ sender, actionType }, 'Rule already exists for sender, skipping creation');
  try {
    await this.client.moveMessage(message.uid, destination, sourceFolder);
  } catch (err) {
    this.logger.error({ uid: message.uid, err }, 'Failed to move duplicate message');
    const dupResult = this.buildActionResult(message, `duplicate-${actionDef.ruleAction}`, duplicate.id, destination, false);
    this.activityLog.logActivity(dupResult, message, duplicate, 'action-folder');
    return { ok: false, action: actionType, error: 'Message move failed' };
  }
  const dupResult = this.buildActionResult(message, `duplicate-${actionDef.ruleAction}`, duplicate.id, destination, true);
  this.activityLog.logActivity(dupResult, message, duplicate, 'action-folder');
  return { ok: true, action: actionType, sender, ruleId: duplicate.id };
}
```

**Change 3 (D-07): Diagnostic logging**

Add a structured pino log call AFTER sender extraction succeeds but BEFORE any business logic (after line 48, before line 50 in current code). Use `logger.info`:

```typescript
this.logger.info({

```

This goes to pino only (not activity log) per discretion -- it's ops debugging, not user-visible audit trail.
