# Requirements: Mail Manager

**Defined:** 2026-04-11
**Core Value:** Dramatically reduce inbox volume without losing visibility

## v0.4 Requirements

Requirements for milestone v0.4: Extended Matchers and Behavioral Learning.

### Extended Matchers

- [ ] **MATCH-01**: System auto-discovers the envelope recipient header by probing common headers (Delivered-To, X-Delivered-To, X-Original-To, X-Resolved-To, Envelope-To) on a sample of recent messages, storing the found header name in config
- [ ] **MATCH-02**: Auto-discovery triggers automatically on successful IMAP connect when server details change, and can be manually invoked from the UI
- [ ] **MATCH-03**: User can create rules that match on envelope recipient using glob syntax (same as sender matching), including +tag variants and catch-all addresses
- [ ] **MATCH-04**: User can create rules that match on header visibility (direct, cc, bcc, list) as a multi-select field, derived by comparing envelope recipient against To/CC fields and checking List-Id headers
- [ ] **MATCH-05**: User can create rules that match on read status (read/unread) at evaluation time
- [ ] **MATCH-06**: When envelope recipient header is not configured (auto-discovery found nothing), envelope recipient and header visibility match fields are disabled and rules using them are skipped

### Behavioral Learning

- [x] **LEARN-01**: System periodically scans Inbox and Review folders using UID snapshot diffing to detect user-initiated message moves, cross-referencing the activity log by Message-ID to exclude system-initiated moves
- [x] **LEARN-02**: For each detected user move, system logs sender, envelope recipient, mailing list headers, subject, read status, header visibility, source folder, and destination folder to a move_signals table
- [x] **LEARN-03**: System performs statistical analysis on move signals to identify repeating patterns (same sender or domain routed to same destination above configurable thresholds)
- [x] **LEARN-04**: System surfaces detected patterns as proposed rules in the UI with approve, modify, or dismiss actions
- [x] **LEARN-05**: Approved proposed rules become real rules in the active ruleset, integrated with existing config hot-reload

### UI Updates

- [x] **UI-01**: Rule editor surfaces envelope recipient glob input, header visibility multi-select (direct/cc/bcc/list), and read status toggle for new match fields
- [x] **UI-02**: Proposed rules view displays detected patterns with signal count, confidence, example messages, and approve/modify/dismiss controls
- [x] **UI-03**: IMAP settings page shows discovered envelope recipient header and provides a button to re-run auto-discovery

## Out of Scope

| Feature | Reason |
|---------|--------|
| Rule effectiveness dashboard | Cut from v0.4 scope -- benefit not worth the cost at this stage |
| User alias configuration | Replaced by auto-discovery of envelope recipient header |
| Move tracking beyond Inbox + Review | Deferred to Tier 6 when action folders add more monitored folders |
| LLM classification | Tier 7, future milestone |
| Auto-apply learned rules (no user approval) | Anti-feature -- system must never change routing without explicit user confirmation |
| CONDSTORE/QRESYNC for move detection | Research confirmed these track flag changes only, not cross-folder moves |
