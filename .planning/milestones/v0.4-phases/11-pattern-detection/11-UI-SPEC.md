## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | #f5f5f5 | Page background |
| Secondary (30%) | #ffffff | Proposal cards, modal surfaces, table backgrounds |
| Accent (10%) | #2563eb | Approve button, active nav tab highlight, toggle checked state |
| Destructive | #dc2626 | Dismiss button text (not bg -- see below), error toast |

Accent reserved for:
- **Approve button** (`.btn-primary` background)
- **Active nav tab** border/background highlight
- **Toggle checked state** on any future toggles in this view

Destructive reserved for:
- **Dismiss button** -- styled as outlined/text-only (not filled red background), to reduce visual weight. Dismiss is reversible (resurface after 5 signals), so it does not warrant a high-alarm destructive fill.

## Component Inventory

### Proposal Card (`.proposal-card`)

Layout: vertical stack inside a white card (same `.settings-card` pattern).

```
+----------------------------------------------------------+
| [Strength Badge]                          [source folder] |
|                                                          |
| sender@domain.com  -->  Destination/Folder               |
| Envelope: recipient@domain.com                           |
|                                                          |
| Recent examples:                                         |
|   - "Subject line one" (Apr 10)                          |
|   - "Subject line two" (Apr 8)                           |
|   - "Subject line three" (Apr 5)                         |
|                                                          |
| [Conflict annotation if applicable]                      |
|                                                          |
| [Approve]  [Modify]  [Dismiss]                           |
+----------------------------------------------------------+
```

**Strength badge:** Inline pill (`.status-badge` pattern) with semantic color per strength tier. Text-only, no numeric scores per D-05.

**Sender -> Destination line:** Sender in regular weight, right arrow (`\u2192`), destination in semibold. Both at body size (14px).

**Envelope recipient:** Shown below sender line at label size (13px), color #666. Omitted if null.

**Example subjects:** Ordered list, max 3 items. Subject text truncated at 60 characters with ellipsis. Date shown in parentheses as short format (e.g., "Apr 10"). Label size (13px), color #444.

**Conflict annotation (D-06):** Only shown when `contradicting_count > 0`. Text: "Also moved to: {folder} ({N}), {folder} ({N})". Label size (13px), color #854d0e on #fef9c3 background (amber warning style).

**Resurfaced notice (D-09):** When a proposal was previously dismissed and resurfaced, show: "Previously dismissed -- {N} new moves since then." Amber warning style matching conflict annotation.

**Action buttons:** Row of 3, right-aligned, 8px gap.
- **Approve:** `.btn .btn-primary` (blue filled)
- **Modify:** `.btn` (default outlined)
- **Dismiss:** `.btn` with color #dc2626, border-color #dc2626, background transparent (outlined destructive, not filled)

### Proposed Page Layout

```
+----------------------------------------------------------+
| Proposed Rules                        [sort: Strongest v] |
+----------------------------------------------------------+
| [Proposal Card 1]                                        |
| [Proposal Card 2]                                        |
| [Proposal Card 3]                                        |
| ...                                                      |
+----------------------------------------------------------+
```

**Toolbar:** Reuse existing `.toolbar` pattern. Left: page heading "Proposed Rules" at heading size (18px/600). Right: sort dropdown (optional -- Claude's discretion per A5, default sort is strength DESC then recency DESC).

**Card list:** Vertical stack of proposal cards with 16px gap between cards.

**Badge count on nav tab (A6):** Show a small count badge on the "Proposed" nav button when active proposals exist. Badge style: 10px font, #fff on #2563eb, border-radius 50%, min-width 18px, centered text. Positioned as inline element after the button text.

### Modify Flow

Modify does NOT open a new modal. It calls the existing `openRuleModal()` function (from app.ts) pre-filled with:
- `sender` field: proposal's sender value
- `deliveredTo` field: proposal's envelope_recipient (if present)
- `action.type`: "move"
- `action.folder`: proposal's destination_folder

On successful save from the rule modal, the frontend calls `POST /api/proposed-rules/:id/approve` to mark the proposal as approved. This requires threading the proposal ID through the modal save callback.

### Empty State

Rendered when `GET /api/proposed-rules` returns an empty array. Uses existing `.empty` class.

### Error State

Rendered when `GET /api/proposed-rules` fails. Uses existing toast pattern (`.toast.error`).

### Loading State

Uses existing `.loading-pulse` pattern while fetching proposals.

| Element | Copy |

| Sort option: recent | "Most recent first" |

### Destructive Actions

| Action | Severity | Confirmation Approach |
|--------|----------|----------------------|
| Dismiss proposal | Low (reversible -- resurfaces after 5 signals) | No confirmation dialog. Immediate action with undo-style toast: "Proposal dismissed." |
| Approve proposal | Medium (creates a real rule) | No confirmation dialog. Immediate action with success toast: "Rule created and active." User can delete the rule from the Rules page if needed. |

Rationale: Neither action is truly destructive. Dismiss is explicitly reversible per D-09. Approve creates a rule that can be deleted. Adding confirmation dialogs would slow down the primary workflow of reviewing proposals.

## Interaction States

### Proposal Card States

| State | Visual Treatment |
|-------|-----------------|
| Default | White card, normal shadow |
| Approve in-flight | Approve button shows spinner (`.spinner` from existing CSS), other buttons disabled |
| Approve success | Card fades out (200ms opacity transition), removed from DOM, success toast shown |
| Dismiss in-flight | Dismiss button shows spinner, other buttons disabled |
| Dismiss success | Card fades out (200ms opacity transition), removed from DOM, dismiss toast shown |
| Modify click | Opens rule editor modal pre-filled. Card remains visible behind modal overlay. |
| Modify save success | Card fades out (same as approve success), rule created toast shown |

### Nav Badge

| State | Visual Treatment |
|-------|-----------------|
| 0 active proposals | No badge shown |
| 1+ active proposals | Count badge (e.g., "3") shown inline after "Proposed" text |
| After approve/dismiss | Badge count decrements in real-time (re-fetch proposals list) |
