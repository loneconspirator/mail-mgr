## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| IMAP message fields -> SQLite | Untrusted email headers (sender, subject, list-id) written to move_signals table |
| YAML config -> runtime | User-editable config values for scanInterval and enabled |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-10-01 | Tampering | SignalStore.logSignal() | mitigate | All SQL uses parameterized queries via db.prepare(...).run(...) with ? placeholders -- no string concatenation of message fields. Verified pattern from existing ActivityLog. |
| T-10-02 | Tampering | moveTrackingConfigSchema | mitigate | Zod schema validates scanInterval as z.number().int().positive() -- rejects non-numeric, negative, or zero values. enabled validated as z.boolean(). |
| T-10-03 | Information Disclosure | SignalStore queries | accept | move_signals stores email metadata (sender, subject) in local SQLite. Single-instance app with no auth boundary -- same trust level as existing activity table. No PII beyond what activity already stores. |
</threat_model>
