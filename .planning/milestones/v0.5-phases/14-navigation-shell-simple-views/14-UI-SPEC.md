### Navigation Architecture

The existing nav uses `data-page` attributes on `<button>` elements in the header `<nav>`. The `navigate()` function switches pages by clearing `#app` and calling the appropriate render function.

Phase 14 adds two new pages to this navigation:
- `data-page="priority"` -> renders Priority Senders view
- `data-page="blocked"` -> renders Blocked Senders view

The existing "Rules" page (NAV-02) continues to show ALL rules unchanged.

### Sender List Table Layout

| Column | Content | Width |
|--------|---------|-------|
| Sender | Sender pattern from rule match (e.g., `*@example.com`) | auto (flex) |
| Rule Name | Optional rule name, muted if absent | 200px max |

Priority Senders table: sender pattern only (action is implicit -- "leave in inbox").
Blocked Senders table: sender pattern only (action is implicit -- "delete").

No action buttons in Phase 14 (inline management is Phase 16).

## Interaction Contracts

### Tab Navigation (NAV-01)

1. User clicks "Priority" or "Blocked" nav button in header
2. Active state transfers to clicked button (`.nav-btn.active`)
3. `#app` clears and renders the appropriate disposition view
4. Fetches from `GET /api/dispositions?type={skip|delete}` on each navigate
5. Loading state: `<p>Loading...</p>` (existing pattern)

### Priority Senders View (VIEW-01)

- Calls `GET /api/dispositions?type=skip`
- Displays sender-only rules where action is `skip` (leave in inbox)
- Each row shows: sender pattern, optional rule name
- Empty state when no results

### Blocked Senders View (VIEW-02)

- Calls `GET /api/dispositions?type=delete`
- Displays sender-only rules where action is `delete`
- Each row shows: sender pattern, optional rule name
- Empty state when no results

### States

| State | Visual |
|-------|--------|
| Loading | `<p>Loading...</p>` centered in `#app` |
| Data loaded | Table with sender rows |
| Empty | `.empty` div with heading + guidance copy |
| Error | `.empty` div with error message (existing pattern) |
