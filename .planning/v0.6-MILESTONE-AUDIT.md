## Cross-Phase Integration

### Verified E2E Flows

1. **VIP Sender** — message to VIP folder -> sender extracted -> skip rule created -> message to INBOX -> folder emptied. All cross-phase links (config -> registry -> processor -> poller -> IMAP) wired.
2. **Block Sender** — message to Block folder -> delete rule created -> message to Trash -> folder emptied. Full chain verified.
3. **Undo VIP** — message to Undo VIP folder -> matching skip rule removed -> message to INBOX. findSenderRule correctly routes through sender-utils.
4. **Duplicate Processing** — same message twice -> no duplicate rule -> activity logged (Phase 23 addition). PROC-07 + LOG-01 wired end-to-end.
5. **Startup Sequence** — ensureActionFolders -> poller.scanAll() -> poller.start() -> monitor.start(). FOLD-01, FOLD-03, MON-02 ordering enforced at index.ts:287-314.
6. **Folder Rename** — Settings UI -> POST /api/folders/rename -> ImapClient.renameFolder -> FolderCache.refresh -> frontend clearFolderCache. Full chain wired.
7. **Config Change Rebuild** — onActionFolderConfigChange handler stops poller, rebuilds with new config, restarts. Code-complete but unreachable from web UI (disk-only config changes).

### Integration Issues (Non-Blocking)

1. **Frontend hardcoded prefix** (`app.ts:1661`): Rename guard uses literal `'Actions'` while backend reads from config dynamically. If `actionFolders.prefix` is changed in config, frontend allows rename attempt that backend correctly blocks with 403. Confusing UX but not a data integrity issue.

2. **No web API for action folder config**: `updateActionFolderConfig` exists in ConfigRepository but has no API route. `onActionFolderConfigChange` handler is wired but unreachable from the web UI. Config changes require editing the YAML file on disk. This is consistent with the requirement text ("configurable" — not "web-UI configurable") but limits operational convenience.
